## 4. MCP server config — `.codex/config.toml` TOML schema

### Stdio server (minimal)

```toml
[mcp_servers.<name>]
command = "npx"
args = ["-y", "@some/mcp-server"]
```

### Stdio server with environment variables

Two forms for environment injection:

```toml
[mcp_servers.<name>]
command = "node"
args = ["path/to/server.js"]
env_vars = ["MY_KEY"]          # inherit named env vars from parent env (pass-through)
env = { MY_KEY = "literal" }   # literal key=value pairs
```

### HTTP / streamable-HTTP server

```toml
[mcp_servers.<name>]
url = "https://mcp.example.com/mcp"
bearer_token_env_var = "MY_TOKEN_ENV_VAR"
http_headers = { "X-Custom" = "value" }
env_http_headers = { "X-Dynamic" = "ENV_VAR_NAME" }
```

### Additional known fields (from migration tooling and reference docs)

- `enabled` (bool) — `false` disables the server without removing it
- `cwd` — working directory for the server process (stdio only)
- `enabled_tools` / `disabled_tools` — tool filtering
- `startup_timeout_sec` — default 10
- `tool_timeout_sec` — default 60
- `bearer_token_env_var` — for HTTP servers with bearer auth
- `env_vars` — list of env var names to inherit from parent process

### Is `required` supported?

**NOT FOUND** in any reference material, help text, migration tooling, or
documentation. The baseline assumed `required = true` as a key — this is
**UNVERIFIED as a real codex config key**. It does not appear in the migration
schema, the Figma MCP reference, or the migration difference docs.

### Project-scoped vs user-scoped config

Both `~/.codex/config.toml` (user) and `.codex/config.toml` (project root) are
supported. The MCP server schema is the same for both. Project config is only
read for **trusted** projects (see §5).

## 5. Project trust mechanism

### Exact mechanism — NOT interactive-only

Project trust is set via `~/.codex/config.toml` using this TOML structure:

```toml
[projects.'<absolute-path>']
trust_level = "trusted"
```

**Examples observed in the real `~/.codex/config.toml`:**

```toml
[projects.'C:\Users\Nova_\PROJECTS\Nova_v0.4.0']
trust_level = "trusted"

[projects.'c:\project-toad']
trust_level = "trusted"
```

### How trust is set non-interactively

**Option A — Direct config file edit:**  
Write `[projects.'<path>'] trust_level = "trusted"` into `~/.codex/config.toml`
directly (pre-write the file before running).

**Option B — `-c` flag override at exec time:**  
```
codex exec -c 'projects."<path>".trust_level="trusted"' --json ...
```
The `-c`/`--config` flag supports dotted-path overrides parsed as TOML, so
nested keys work. Note: path separator quoting may be tricky on Windows with
backslashes.

**Option C — `codex mcp add` command** (for MCP servers specifically):  
`codex mcp add <name> -- <command> [args...]` adds a server to the user config,
but does not directly set project trust.

**Conclusion: project trust has a clear non-interactive config path.** Writing
`trust_level = "trusted"` into `~/.codex/config.toml` before launching is fully
headless. This is NOT design-blocked.

**`c:\project-toad` is already present** in `~/.codex/config.toml` with
`trust_level = "trusted"`.
